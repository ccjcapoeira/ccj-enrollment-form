# GitHub公開リポジトリ用 フォーム安全確認チェックリスト

このチェックリストは、`ccj-enrollment-form` のような **個人情報を扱う申込フォーム** を GitHub リポジトリ上で公開・運用する際に、危険な公開や見落としを防ぐための確認用メモです。

目的は、**「HTML を GitHub に置くこと」自体ではなく、「公開してよいもの」と「公開してはいけないもの」が混ざっていないかを確認すること**です。

---

## 前提整理

GitHub Pages や公開 GitHub リポジトリで、以下を公開すること自体は一般的です。

- HTML
- CSS
- フロントエンド JavaScript
- 画像
- UI 文言
- クライアント側バリデーション

ただし、以下が公開されていると危険です。

- API キー
- Secret / Token / Password
- Webhook URL
- Google Apps Script の実行 URL（秘匿前提で使っている場合）
- 管理用メールアドレスや内部運用情報
- 個人情報の保存先にアクセスできる認証情報
- 第三者に悪用されうる送信エンドポイント情報

---

## Cursor への依頼内容

以下を順番に確認し、必要なら修正してください。

### 1. 公開リポジトリに秘密情報が含まれていないか確認

確認対象:

- ソースコード全体
- 設定ファイル
- 環境変数ファイル
- ビルド設定
- デプロイ設定
- Git の履歴

重点チェック:

- `.env`
- `.env.local`
- `config.*`
- `settings.*`
- `appsscript.json`
- `firebase config`
- `netlify.toml`
- `vercel.json`
- JavaScript / TypeScript / HTML 内の直書き URL

探してほしい文字列例:

- `apiKey`
- `secret`
- `token`
- `password`
- `Authorization`
- `Bearer`
- `webhook`
- `script.google.com`
- `smtp`
- `client_secret`
- `private_key`

やってほしいこと:

- 危険な値が見つかったら一覧化
- 本当に秘密情報かどうか判定
- 秘密情報ならリポジトリから除去
- すでにコミット済みなら Git 履歴に残っていないか確認
- 必要ならローテーション対象も明記

---

### 2. フォーム送信先の設計が安全か確認

確認観点:

- フォーム送信先 URL がフロントコードに露出しているか
- 露出している場合、その URL を第三者が悪用できるか
- 認証なしで大量投稿できる構造になっていないか
- スプレッドシートやメール送信に直接つながっていないか
- GAS や外部サービスのエンドポイントが公開前提で安全に設計されているか

チェックしてほしいこと:

- 送信先 URL の特定
- 送信時の認証有無
- CORS 設定
- rate limit 相当の考慮有無
- bot 対策有無
- CSRF 的な問題の有無
- 外部から同じ POST を再現できるか

必要なら提案してほしい改善:

- Cloudflare Turnstile / reCAPTCHA の導入
- honeypot フィールド追加
- 送信間隔制限
- 中継 API の導入
- 機密性のある送信処理をサーバー側へ移動

---

### 3. 個人情報の扱いと実装が一致しているか確認

フォームでは以下のような個人情報を扱う可能性があります。

- 氏名
- フリガナ
- 生年月日
- 電話番号
- メールアドレス
- 住所
- 保護者情報

確認してほしいこと:

- 実際に送信している項目の一覧
- 保存先
- 誰がアクセス可能か
- 利用目的と実装の整合性
- 不要なデータを取得していないか
- ログや通知に個人情報が過剰に残っていないか

見てほしいポイント:

- ブラウザコンソールに個人情報を出力していないか
- analytics / error tracking に個人情報が流れていないか
- query parameter に個人情報を載せていないか
- サンクスページ URL に個人情報が入っていないか
- メール通知本文に過剰な個人情報が含まれていないか

---

### 4. Git の履歴に危険情報が残っていないか確認

重要:

今のファイルから秘密情報を削除していても、**過去のコミット履歴に残っていれば漏えいリスクは継続**します。

やってほしいこと:

- 現在のブランチだけでなく履歴も確認
- 過去コミットに API キーや webhook URL がないか確認
- 見つかった場合、失効・再発行が必要か判断
- 必要なら履歴書き換え方法を提案

確認対象例:

- old config files
- deleted files
- 過去の試作コード
- コメントアウトされた秘密情報

---

### 5. 公開してよい内部情報かどうか確認

秘密情報ではないが、公開すると好ましくない情報も確認してください。

例:

- 管理者個人のメールアドレス
- 個人携帯番号
- 内部の運用メモ
- 事務フロー
- 将来使う予定の非公開 URL
- 管理画面 URL
- テストデータに含まれる実在個人情報

必要なら以下を提案してください。

- 問い合わせ先表記の整理
- テストデータのダミー化
- コメント削除
- README の公開向け整備

---

### 6. スパム・荒らし・誤送信への耐性確認

このフォームは公開 URL で誰でもアクセスできる想定なので、以下を確認してください。

- bot による大量送信のしやすさ
- 同一内容の連投対策
- 必須項目だけ満たした機械投稿への耐性
- メール通知爆撃が起きないか
- スプレッドシート破壊的入力のリスク
- HTML やスクリプト文字列の取り扱い

確認してほしいこと:

- 入力値サニタイズ
- 改行やタグの扱い
- 長文制限
- 電話番号やメール形式の検証
- サーバー側再検証の有無

改善提案候補:

- honeypot
- invisible captcha
- submission cooldown
- IP / UA ベースの簡易制御
- 管理通知の頻度制御

---

### 7. フロントエンドだけで完結していることによる限界の整理

もし現在のフォームが静的サイト + 外部送信先だけで動いている場合、以下を整理してください。

- どこまでがクライアント側防御か
- どこから先はサーバー側でないと守れないか
- 現状構成で十分な点
- 将来的にバックエンド化した方がよい点

とくに知りたいこと:

- 現状の構成は「小規模教室の申込フォーム」として許容範囲か
- どの条件なら次の段階のセキュリティ対策が必要か
- コストを抑えた現実的改善案は何か

---

## 出力してほしい形式

以下の形式で整理してください。

### A. 危険度サマリー

- High
- Medium
- Low

### B. 発見事項一覧

各項目について:

- 問題の内容
- なぜ危険か
- 現状の影響範囲
- いますぐ直すべきか

### C. 修正提案

- すぐできる修正
- 中期的にやるとよい修正
- 必要なら構成変更が必要な修正

### D. 実際の修正

可能ならコードレベルで修正してください。

### E. 残課題

- 今は対応しなくてもよいが、将来的に検討すべき点

## 追加の希望

- 既存構成を大きく壊しすぎずに確認してほしい
- 単なる一般論ではなく、このリポジトリの実装を見て判断してほしい
- 問題がなければ「大きな問題なし」と明記してほしい
- 問題がある場合は、推測ではなく根拠ベースで示してほしい
- 可能なら、初心者にも分かる表現で説明を添えてほしい

## Cursor にそのまま渡す用の短い依頼文

このリポジトリは GitHub Pages で公開している入会申込フォームです。  
公開リポジトリに置くこと自体の問題ではなく、**秘密情報の混入、送信先の安全性、スパム耐性、個人情報の取り扱いの整合性、Git 履歴漏れ**がないかを重点的に監査してください。  
`github_form_security_checklist_for_cursor.md` の観点に沿って確認し、危険度サマリー、発見事項、修正提案、可能なら実コード修正まで整理してください。
